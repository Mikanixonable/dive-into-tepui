## ステップ 0: 計測基盤の整備(最初にやる・必須)

軽量化は「測ってから削る」。以下を先に用意する。

- Chrome DevTools Performance / Memory(Allocation sampling)で 1 分間のプレイを記録し、`Game.update` 内の関数別 self-time と GC 頻度のベースラインを取る。
- `?stage=` パラメータと同様の URL パラメータ(例 `?perf=1`)で、簡易フレームタイム表示(sim 時間 / render 時間 / entity 数)を HUD に出すデバッグ表示を追加する(これ自体は挙動を変えない)。
- 高負荷の再現シナリオを固定する: ステージ 00(サバイバル・ウェーブ)で敵・弾・薬莢・デブリが最大化した状態、および高倍率タイムワープ(サブステップ数最大)の 2 つ。
- 変更ごとに `npm run typecheck` と `npm run test:physics` を回す(特に `src/physics/` を触る場合は必須)。

## カテゴリ A: 仕様を変えないリファクタリング的軽量化

見た目・物理挙動・操作は一切変えず、計算の無駄と GC 圧力を削る。

### A-1. Vec3 演算のアロケーション削減 【影響度: 大 / リスク: 中】

- 現状: `vec3.ts` の `add`/`cross`/`norm` 等は毎回新しい `{x,y,z}` を返す。`game.ts` だけで 87 呼び出し箇所があり、サブステップ×エンティティ数(敵・弾・薬莢・デブリ)×RK4 の 4 ステージで毎フレーム数千〜数万個の短命オブジェクトが生成される。
- 施策: `vec3.ts` に out 引数版(`addTo(out, a, b)` 等)を追加し、ホットパス(`stepOrbitRK4`、`makeEnvAccel` の drag+J2+第三体合成、`stepAttitude`、`belt.ts` の PBD ループ)から順に置き換える。既存 API は残すので非ホットパスは触らない。
- 検証: `test:physics` が数値的同一性をそのまま担保する。
- 注意: 純粋関数という設計思想を壊さないよう、out 版はモジュール内スクラッチに限定し、公開 API の意味は変えない。

### A-2. 毎フレームの `new THREE.*` 排除(スクラッチ再利用) 【影響度: 中 / リスク: 小】

- 現状: `belt.ts`(23 箇所)ほか、フレーム毎に呼ばれる経路に `new THREE.Vector3`/`Quaternion` 生成が残る。
- 施策: モジュールスコープ / インスタンスフィールドのスクラッチベクトルに置き換える(`belt.ts` は既に「per-frame scratch vectors」方針があるので徹底する)。`camera.ts`・`mapview.ts`・`markers.ts` も同様に確認。

### A-3. 環境加速度の呼び出し内冗長計算の共通化 【影響度: 中 / リスク: 小】

- 現状: `makeEnvAccel` は RK4 の各ステージで J2 + 第三体 + 大気を都度計算する。太陽・月の位置はサブステップ内では実質不変なのに、`thirdBodyAccel` へ渡す位置の再取得や中間 Vec3 生成が毎ステージ発生する。
- 施策: サブステップ開始時に太陽・月位置・大気の高度依存項をキャッシュし、クロージャで参照する。数値結果は同一(すでに「フレーム内固定」の前提が成立している範囲のみ)。

### A-4. HUD の DOM 更新削減 【影響度: 中 / リスク: 小】

- 現状: `hud.ts` の `setTarget`/`setEnemyList` 等が `innerHTML` を再構築する(l.204〜227 付近)。`setPlanPanel` は差分チェック済み(l.245)だが他は毎回書き換えになり得る。`markers.ts`(HUD 側)は SVG リーダー線の `setAttribute` を毎フレーム発行。
- 施策: すべての `innerHTML` 系更新に「前回文字列と同一ならスキップ」を徹底し、頻度の低い情報(敵リスト、ターゲット詳細)は `markers.ts`(game 側)が既に持つリフレッシュタイマーの間引きに揃える。SVG 属性は値が変わったときだけ set。
- 注意: レイアウト/リフローは WebGPU と別軸でフレームを食うため、DevTools の "Rendering → Paint flashing" で効果を確認する。

### A-5. `resolvePhysicalCollisions` の近傍絞り込み 【影響度: 中 / リスク: 中】

- 現状: 薬莢全体に対する O(n²) の球衝突。`MAX_PHYS_WARP` 以下でのみ実行という緩和は既にあるが、薬莢が溜まると効く。
- 施策: 判定対象をプレイヤー近傍(例: 既存の clank 判定距離のような数十 m 半径)に事前フィルタする、または簡易グリッド/距離ソートで対を間引く。遠方の薬莢同士の衝突は現状でも実質発生しない(仕様上の観測可能な変化なし)ことを確認したうえで導入する。

### A-6. 軌道線・予測線ジオメトリの再利用 【影響度: 小〜中 / リスク: 小】

- 現状: `orbitline.ts` は固定長 `Float32Array` 再利用で良い設計だが、`trajline.ts` は再構築のたびに `new Float32Array` + 新 BufferAttribute を生成する(l.33-41)。
- 施策: `trajline.ts` を最大サンプル数(predict の `maxSamples` 既定 2000)で確保した固定バッファ + `setDrawRange` 方式に変更。GPU への再アップロードは dirty 時のみ。

### A-7. 死んだエンティティの配列管理 【影響度: 小 / リスク: 小】

- 現状: `bullets`/`casings`/`debris` を `alive` フラグ付きで全走査する箇所が多い(game.ts l.939-1068 ほか)。
- 施策: 除去時 swap-pop の徹底とループの統合(同じ配列を複数回走査している箇所をまとめる)。挙動不変。

**カテゴリ A の期待効果**: GC 起因のスパイク解消(体感のカクつきに最も効く)+ フレームあたり CPU の 20〜40% 削減が目安。

## カテゴリ B: 仕様に影響するが効果の大きい軽量化

見た目や精度がわずかに変わる。影響度=「プレイヤーが気づく可能性」で区分。

### B-1. 遠方・非重要エンティティの物理 LOD 【効果: 大 / 仕様影響: 小】

- 薬莢・デブリはプレイヤーから一定距離(例 500 m)を超えたら RK4 → 単純ケプラー伝播(または積分頻度を 1/4 に間引き)へ格下げし、姿勢積分(`stepAttitude`、ジャニベコフ効果)も視認距離外では停止する。
- 画面上で判別できない距離のみ対象なので実質的な見た目の変化はないが、「全エンティティが常に厳密軌道」という現仕様からは外れる。

### B-2. 薬莢・デブリの上限と寿命 【効果: 大 / 仕様影響: 小〜中】

- 現状は無制限に蓄積し得る。上限数(例 薬莢 200 / デブリ 300)と寿命(例 120 s でフェードアウト消滅)を導入。長時間戦闘での「過去の薬莢が全部軌道上に残る」仕様は失われるが、CPU・描画双方の最悪ケースを確定的に抑えられる。

### B-3. 大気シェルの削減または動的品質設定 【効果: 大(GPU) / 仕様影響: 中】

- 24 枚の加算ブレンドシェルは水平線付近で大きなオーバードローになる。枚数を高度別に統合(例 16+8 → 10+5)し、opacity を再調整して見た目を近づける。あわせて設定メニューに品質段階(高/中/低: シェル枚数・地球球の分割数 512×384 → 256×192・星の数)を追加する。
- 見た目の変化はあるが、既存の「離散バンドが見えない」制約を検証しながら詰める。

### B-4. HUD・マーカー更新レートの間引き 【効果: 中 / 仕様影響: 小】

- スクリーン投影マーカー(markers.ts)とナビボール類の再描画を 60fps → 30fps(2 フレームに 1 回)に間引く。照準・リードマーカーなど戦闘直結のものは 60fps を維持し、AN/DN・敵リスト・ステータスパネルのみ間引く。マーカーのわずかな追従遅れが生じる。

### B-5. 衝突判定・AI の更新間引き 【効果: 中 / 仕様影響: 中】

- `checkBoardCrossings` や敵 AI(`updateEnemyAI`)を毎サブステップ → 毎 simulate 呼び出し 1 回に間引く。弾のトンネリング対策(線分 vs 球)は既にあるため命中判定の正確性は保てるが、AI 反応のタイミングがわずかに粗くなる。

### B-6. 物理のワーカースレッド化(将来・大工事) 【効果: 大 / 仕様影響: 実装リスク大】

- 既存の未使用 N-body ワーカー基盤を活かし、薬莢・デブリなどゲームプレイに非干渉なエンティティ群の積分をワーカーへ移す。メインスレッドは描画 + プレイヤー/敵/弾のみ。1 フレーム遅延の位置反映という仕様変化を伴う。cislunar フェーズの布石にもなるが、着手は A 系と B-1〜B-3 の効果測定後に判断する。

## 実施順序(推奨)

| 順 | 項目 | 区分 | 理由 |
|---|---|---|---|
| 0 | 計測基盤 | — | 効果検証の前提 |
| 1 | A-1, A-2(アロケーション削減) | 仕様不変 | GC スパイクが体感に最も効く。test:physics で安全に検証可能 |
| 2 | A-4, A-6(DOM / ジオメトリ) | 仕様不変 | 低リスクで確実 |
| 3 | A-3, A-5, A-7 | 仕様不変 | 中リスク。計測で効果があるものだけ |
| 4 | B-2(上限・寿命) | 仕様変更・小 | 最悪ケースの確定的な抑制 |
| 5 | B-1(物理 LOD)、B-4 | 仕様変更・小 | 計測でエンティティ積分が支配的なら |
| 6 | B-3(GPU 品質設定) | 仕様変更・中 | GPU バウンドと判明した場合 |
| 7 | B-5, B-6 | 仕様変更・大 | 上記で不足する場合のみ |

各ステップの完了条件: 固定シナリオ(ステージ 00 高密度・高倍率ワープ)でのフレームタイム改善を記録し、`typecheck` + `test:physics` 通過、および見た目・挙動の目視確認(B 系は変化点を明記してコミット)。
