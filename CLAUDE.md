# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Current state

A playable LEO (low Earth orbit) shooting game: TypeScript + Webpack + npm, Three.js `WebGPURenderer` (via the `three/webgpu` entry point). Real-scale/real-time Earth orbit, KSP-style frame-based RCS translation (orbital / target reference frames), manual rotation with RCS damping, time warp, lead-marker gunnery, shell casings and destruction debris on accurate orbital physics, win on destroying all 5 enemies, lose on atmospheric reentry (drag decay → adiabatic-heating overheat or dynamic-pressure breakup). Also: Earth-shadow lighting, a KSP-style navball (canvas 2D, bottom center), target-board bullet-pass markers, an orbit-planning map mode ([M]: Earth-centered camera, click the player orbit to place a maneuver node, W/S/A/D/Q/E adjust Δv, [N] auto-warps to the node, BURN guide shows remaining Δv until the actual orbit matches the planned elements within tolerance), and two stages (stage 2 = Molniya-orbit enemies, unlocked via localStorage `tepui.stage1.cleared`; `?stage=1|2` URL param skips the select screen). The renderer draws `game.activeCamera` — combat chase cam (near 2 / far 6e7) or the map camera (near 1e4 / far 6e8); stars/sun/moon billboards follow the active camera position so the map zoom-out doesn't exit the star shell.

`dev.md` is explicitly marked as human-authored only ("この文書は人間のみが記入できる") — do not edit it. Read it for project context, but leave modifications to the user.

### Commands
- `npm run dev` — start webpack-dev-server at http://localhost:8080
- `npm run build` — production build to `dist/`
- `npm run typecheck` — `tsc --noEmit`

There is no automated test suite wired into npm. Physics can be verified by compiling `src/physics/*.ts` to CommonJS with `tsc` and running assertions in node (pure functions, no DOM/THREE deps).

### Architecture

Simulation state lives in ECI coordinates (Y axis = north pole), SI units (m, m/s), as plain `{x,y,z}` data. Rendering uses a **floating origin**: the player ship is always at world (0,0,0) and everything else (including the Earth mesh at `-playerR`) is positioned relative to it each frame, so f32 GPU precision never sees absolute LEO magnitudes. Camera: near=2m, far=6e7m — no logarithmic depth needed (see comment in `scene.ts`).

Physics runs on the main thread (per-entity central-gravity two-body integration is cheap); the N-body worker is **currently unused**, kept for the future cislunar (Sun-Earth-Moon) phase.

- `src/main.ts` — entry point; WebGPU init, error overlay fallback, rAF loop driving `Game.update`.
- `src/game/game.ts` — orchestrator: entity management, substepped integration (higher warp → more substeps, max 20s each), input→thrust/torque, segment-vs-sphere bullet collision (tunneling-proof), win/lose, render sync, HUD markers (boresight/lead/prograde/target). `autoAlignTorque` implements the [C] prograde-hold autopilot: builds a desired attitude quaternion (nose = velocity direction, roll reference = radial-out, re-orthogonalized so it's valid even for eccentric orbits where r⟂v doesn't hold exactly), converts the quaternion error to a world-frame axis-angle, rotates the axis into body frame (torque must be expressed in body frame — that's what `stepAttitude`/Euler's equations expect), and applies a PD torque (`PROGRADE_HOLD_KP`/`_KD`).
- `src/game/const.ts` — all gameplay tuning constants (thrust, warp levels, fire rate, hit radii...).
- `src/game/navball.ts` — canvas-2D attitude ball (body frame: +X right/+Y up/+Z nose); sphere is repainted per frame via a per-pixel dot product against the body-frame Earth direction (cheap: rotate Earth dir into body frame once, not per pixel).
- `src/physics/atmosphere.ts` — piecewise-exponential density model (Vallado table, 0–1000 km). Drag (Cd·A/m per entity class, co-rotating atmosphere) is applied as the RK4 extra-acceleration; player hull temp uses Sutton–Graves stagnation heating + Stefan-Boltzmann radiative cooling (see `updateThermal` in game.ts).
- `src/game/{input,camera,hud,audio,entities}.ts` — keyboard/mouse state + edge-trigger queue (left button = camera drag, with a short-drag click queue for map-mode node placement; right button = fire); ship-centered chase camera (up = radial), also driven by arrow keys via `C.CAM_KEY_YAW_RATE`/`CAM_KEY_PITCH_RATE`; DOM-overlay HUD — **flat dark theme: monotone grays on near-black panels (1px light edge, no shadows) with one saturated orange accent** (`#ff6a00`); markers and orbit lines follow the same rule — monotone base (player orbit gray, planned orbit white, prograde/boresight light gray) with orange reserved for attention targets (target marker+orbit, lead, maneuver node/BURN, ammo pickups); WebAudio synth SFX (CIWS-motif: motor-whine spin-up, heavy discrete gun reports) + Steve-Reich-style ambient minimal BGM — modal quartal pitch set (neither major nor minor), two pulse patterns of coprime lengths (16 vs 12 steps) that phase against each other, slow quartal pads and a deep drone, sine/triangle only (no assets; starts on first user gesture, stops on win/lose); entity type defs.
- `src/physics/orbital.ts` — central-gravity RK4 for one entity with optional extra-acceleration callback (thrust, evaluated per RK4 stage), state→orbital elements, ellipse sampling for orbit lines, `j2Accel` (Earth oblateness, Y = polar axis; verified ≈ −5°/day RAAN regression at 420 km/51.6°), `thirdBodyAccel` (differential tidal accel), and Kepler helpers for maneuver planning (`trueAnomalyAt`, `tofBetween`, `positionOnOrbit`/`velocityOnOrbit`, `stateFromElements` — all round-trip-verified to machine precision). Pure functions.
- `src/physics/ephemeris.ts` — circular-orbit Sun/Moon ephemerides in game ECI (Y = north): Sun on the ecliptic (23.44° obliquity), Moon inclined 5.145° to the ecliptic with 18.61-year regressing node (equatorial inclination sweeps 18.3–28.6°, which drives real inclination perturbation of LEO orbits). All entities integrate drag + J2 + lunisolar third-body via `makeEnvAccel` in game.ts.
- `src/physics/attitude.ts` — rigid-body attitude: quaternion + body-frame ω via Euler's equations. ω integrated with RK4 + kinetic-energy projection (naive explicit integration diverges — this is what makes the Dzhanibekov effect on debris stable long-term). Pure functions.
- `src/physics/vec3.ts` — plain-object Vec3 math helpers.
- `src/physics/{bodies,integrator,physics.worker}.ts` — original N-body RK4 worker stack, unused by the LEO game; retained for the cislunar phase.
- `src/render/earth.ts` — realistic-style Earth: high-res indexed `SphereGeometry` (512×384) with smooth per-vertex colors from deterministic 3D fBm noise (continuous smoothstep-blended biomes, no facet jitter), clouds **baked into vertex colors** (a separate cloud shell z-fights with the surface near the horizon at 24-bit depth), baked cloud shadows (offset eastward in the earth-fixed frame — an approximation that doesn't track the sun), stacked additively-blended **Lambert-lit** shells for continuous atmosphere density with a twilight gradient across the terminator — 16 FrontSide + 8 BackSide rim shells, quadratically spaced in altitude with exponentially decaying opacity so no discrete bands are visible; all Lambert (a Basic rim shell stays bright on the night side and reads as a band); max shell ≤ 340 km so the camera stays outside them at normal flight altitude — and wavy aurora curtains (vertex-color fade, additive; animated via `Earth.tick`) fixed to the spin group at ±67° latitude.
- `src/render/ships.ts` — primitive-built low-poly meshes: player ship (nose = body +Z; stubby box hull, 4 aft engines, small solar paddles, two vertical gun bores on the front face — `MUZZLE_OFFSETS` and `RCS_BLOCK_OFFSETS` are exported so game.ts fires/ejects/puffs from the exact modeled positions), magazine belt links (`buildMagazineMesh`: caged 2×4 visible brass rounds, dims tied to `MAG_THICKNESS` = ¼ hull height) and orbital ammo pickups (`buildMagPickup`), enemy variants, shared-geometry tracer bullets, CIWS-motif bottleneck casings (`LatheGeometry` profile: rim + extractor groove + tapered body + shoulder + neck), debris shards, billboard flash.
- `src/render/{stars,orbitline}.ts` — stars as tiny world-space triangles (WebGPU points are 1px; `THREE.Points` size doesn't work), sun billboard and moon mesh — both placed along their true ephemeris directions each frame at compressed display distances (4.2e7 / 4.5e7 m, inside the 6e7 far plane) with the moon's angular size derived from its real distance, so lunar phases come free from the directional sun light; orbit ellipse lines (`THREE.Line` — **`LineLoop` is unsupported by the WebGPU renderer**, close the loop manually). OrbitLine vertices are **Earth-centered ECI**, moved per frame only via `line.position = -origin`; geometry regenerates only when osculating elements drift past thresholds (or `force=true` during burns/node editing) — rewriting vertices in ship-relative coords every frame made the ellipse visibly jitter under J2/third-body element wobble. The target's orbit line is drawn gold at `renderOrder=2` (above the player's cyan line at `renderOrder=1`) — a rendezvous target's orbit is often nearly coincident with the player's, and relying on transparent-object draw order alone made it disappear underneath.
- `src/types/three-shims.d.ts` — three.js 0.169 ships no `.d.ts` for `three/webgpu`; minimal `WebGPURenderer` typings. Remove once upstream types cover it. Import THREE **only** from `'three/webgpu'` (mixing with `'three'` would duplicate classes at runtime).

### WebGPU renderer gotchas (three.js 0.169)
- `THREE.LineLoop` and `THREE.Points` sizing are not supported; use `THREE.Line` / triangle-based sprites.
- Additive-blended materials need explicit `transparent: true`.
- Headless-Chrome screenshots work with `--headless=new --enable-gpu --enable-unsafe-webgpu --disable-gpu-sandbox --no-sandbox` (flaky; retry a few times).

Not yet implemented: enemy AI (targets are passive), solar-storm/combat auroras, ECS, full N-body cislunar phase. See `dev.md` for design direction.

## Project concept (from dev.md)

An orbital-mechanics shooting game, intended as a web game, set in the Earth-Moon system (current build: LEO only):

- Gameplay centers on realistic orbital maneuvers and physically accurate ballistic calculations, low-poly but visually appealing depictions of Earth's atmosphere, spacecraft, and spacecraft destruction.
- Orbital calculations should account for Sun-Earth-Moon three-body dynamics, atmospheric drag, and tidal forces. Reference concepts to consider: Sun-Earth and Earth-Moon Lagrange points, sun-synchronous recurrent orbits, halo orbits, low-energy transfers and ballistic capture, double lunar swingbys, and gradual altitude-raising approach maneuvers (like the Kounotori HTV's ISS approach).
- Aesthetic goals: Earth auroras, auroras from solar storms, auroras from charged particles generated by combat, shadows on Earth's evening clouds, and the Dzhanibekov effect on destroyed debris.

## Controls (current build)

W/S = prograde/retrograde, A/D = normal±, Q/E = radial in/out (F switches to target-relative frame: approach/retreat/left/right/down/up). I/K/J/L/U/O = pitch/yaw/roll, T = RCS rotation damping, C = prograde-hold autopilot (PD controller torquing nose to velocity vector; any manual rotation key cancels it), Tab = cycle target, Space / **right click** = fire, **left drag** or **arrow keys** = camera (both drive the same yaw/pitch — arrow keys are a convenience alias, not a separate control scheme), wheel = zoom distance, `,` / `.` = time warp (thrust and guns locked above ×4), P = pause, H = help, R = restart (after game end).

Ammo system: guns fire alternately from the two front bores and consume magazine rounds (16/mag, 32 mags linked at start). The belt of unused mags hangs off the ship's starboard (+X) side as child meshes of `player.obj`, and the belt slides continuously into the hull as rounds are fired (`beltFeed` = 1 − rounds/16 offsets the anchor by up to one pitch; on mag consumption the feed wraps 1→0 exactly when the link count drops, so the motion reads as seamless). The head link sits half-inside the hull. `Game.updateBeltPhysics` simulates the belt's sag as an actual physics chain, not a canned animation: link joints are point masses (`beltPos`/`beltPrevPos`, Verlet-integrated in the ship's body frame) connected by rigid-rod distance constraints (position-based dynamics, 4 relaxation iterations per frame) to `MAG_BELT_PITCH`. Since the ship is in free fall, ordinary gravity can't sag anything (equivalence principle — belt and hull fall together); the only real driver is the fictitious force felt in the ship's non-inertial (accelerating/rotating) frame: `-a_thrust - α×r - ω×(ω×r) - 2ω×v` (translational inertia, Euler, centrifugal, Coriolis), built from `thrustAccelVec` (world-frame thrust accel, rotated into body frame) and `player.att.w` (with `α` estimated by finite-differencing `w` against `prevBodyW`). When the player isn't thrusting or rotating, the chain just hangs straight — no ambient wobble is faked in.

When mags run low (`AMMO_LOW_MAGS`), resupply pickups spawn phase-shifted 2.5–5 km ahead on the player's own orbit (▣ AMMO marker, absorb within `MAG_PICKUP_RADIUS`). Casings use a CIWS-style bottleneck profile (rim/extractor groove/tapered body/shoulder/neck via `LatheGeometry`) with **low** ejection speed/spin (previous values made them visibly spin sideways at high rate) and drift back within ~3.2 m of the hull play a faint clank — but only after first travelling >6 m away (`clankArmed`), since every casing starts near the hull. Spin-up SFX is filtered-noise friction (no oscillator). Enemy ships are modeled at `ENEMY_SCALE` (10×) via `obj.scale.setScalar(...)` with `ENEMY_RADIUS` (90 m) matching for hit detection; their destruction VFX (`spawnFlash` sizes, `spawnFragments` size range) scale by the same factor so the explosion reads proportionate to the larger hull.

Mouse buttons are swapped from the naive default: **left button drags the camera; a left click that moves less than `CLICK_MOVE_THRESHOLD` (6px) is instead queued as a click** (used to place maneuver nodes in map mode) — see `Input.takeClicks()`. Right button fires. This lets map-mode node placement and camera dragging share the left button without a modifier key.

Map mode ([M]) no longer freezes physics: `simulate()` always runs while `phase==='playing'`, with ship control (thrust/fire/manual attitude) gated by `!this.mapMode` since WASDQE is repurposed for Δv editing there — so time warp keeps working and the player can wait for a transfer window while planning. `updateMapPlanning()` only runs the click/Δv-edit overlay on top. The map camera's drag yaw sign is intentionally opposite the combat chase camera's convention (`+=` vs `-=` on `mouse.dx`) so panning left/right reads the same in both views.
