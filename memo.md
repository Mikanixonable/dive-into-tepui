# Geminiによる実現可能性調査
結論から言うと、**十分に作成可能**です。

現在のブラウザ環境（特にJavaScriptエンジンとWebGL/WebGPU）は非常に強力であり、ハードSF的なアプローチが求められる複雑な軌道力学のシミュレーションをブラウザ上でリアルタイムに動かすことができます。月面開発のロジスティクスや、L1/L2ラグランジュ点を利用した物資輸送シミュレーションなどを表現するには非常に面白いテーマです。

実現に向けてクリアすべき、3つの技術的な柱とその実装方針を解説します。

## 1. 軌道計算（物理エンジン）の要件

L1、L2のようなラグランジュ点周辺の軌道（ハロー軌道やリサジュー軌道）は力学的に非常に不安定（カオス的）です。わずかな誤差があっという間に拡大するため、自作の物理エンジンの精度がゲームのコアになります。

* **太陽の摂動（N体問題）:**
地球、月、宇宙船の制限3体問題に、太陽の重力を加えた「4体問題」として計算します。各天体からの重力加速度は以下の式で合成します。

$$\mathbf{a}_i = \sum_{j \neq i} G \frac{m_j}{\vert{}\mathbf{r}_{ij}\vert{}^3} \mathbf{r}_{ij}$$


* **自転の潮汐力（非球形重力場）:**
自転による天体の歪み（赤道ふくらみ）が軌道に与える影響は、「J2摂動」として実装します。極軌道や低軌道（LEO）での軌道面歳差運動をシミュレートするのに不可欠です。
* **大気抵抗（Atmospheric Drag）:**
L1やL2（地球から約38万km）の距離では完全に無視できますが、地球や月の近傍でのエアロキャプチャーや打ち上げを想定するなら必須です。高度に応じた大気密度モデル（単純な指数関数モデルなど）を用意し、以下の式を組み込みます。

$$F_D = \frac{1}{2} \rho v^2 C_D A$$



**【計算手法の選択】**
単純なオイラー法ではすぐに軌道が破綻します。最低でも**4次のルンゲ＝クッタ法（RK4）**、より長期の安定性を求めるなら**シンプレクティック積分**（Verlet法など）を採用する必要があります。JavaScriptは標準で64ビット浮動小数点数（Double精度）をサポートしているため、位置座標の計算自体の精度はブラウザでも十分に確保できます。

## 2. 推奨される技術スタック

UIの構築や状態管理の観点から、モダンなWebフロントエンド技術を活用するのが最もスムーズです。

| コンポーネント | 推奨技術 | 理由 |
| --- | --- | --- |
| **UI・状態管理** | React + React Three Fiber | 複雑な計器類のUIと、3D空間上のオブジェクトの状態同期が容易なため。 |
| **3D描画** | Three.js (WebGL/WebGPU) | 宇宙空間のライティングや地球の大気シェーダーなどの実装実績が豊富。 |
| **物理演算** | Web Workers (JS/TS) | 重い積分計算を別スレッドに逃がし、UI描画（60FPS）のブロックを防ぐため。 |

※さらに計算速度が必要になった場合は、物理演算部分のみをRust等で記述し、**WebAssembly (Wasm)** としてブラウザに読み込ませる拡張も可能です。

## 3. 開発における最大の壁

ゲームとして成立させる場合、以下の2つの問題に直面します。

1. **タイムワープ機能の実装:**
月まで数日かかるため、ゲーム内時間を1000倍などに加速させる必要があります。単純に1フレームあたりの時間ステップ（$\Delta t$）を大きくすると計算が爆発して宇宙船が地球を突き抜けます。Dormand-Prince法などの「刻み幅制御付きルンゲ＝クッタ法」を用いて、天体に近づいた時だけ自動的に計算の解像度を上げる工夫が必要です。
2. **極端なスケール差（Zファイティング）:**
地球（半径約6,400km）から宇宙船（数メートル）までを描画しようとすると、通常の深度バッファでは精度が足りず、テクスチャがチラつく「Zファイティング」が発生します。Three.jsで `logarithmicDepthBuffer: true` を有効に設定することが必須となります。

このプロジェクト **`dive-into-tepui`** は、**地球低軌道（LEO）を舞台にした、リアルな軌道力学に基づく3Dシューティングゲーム**です。

技術的には、**TypeScript + Three.js（WebGPUレンダラー）**で構築されており、本格的な宇宙シミュレーター（Kerbal Space Programのようなシステム）とシューティングゲームの要素を融合させています。

以下にプロジェクトの主要な特徴、アーキテクチャ、およびコード構成について解説します。

---

### 1. 主な機能と特徴

*   **リアルな軌道力学のシミュレーション**
    *   RK4（ルンゲ・クッタ4次）数値積分を用いて、自機や敵機、薬莢、デブリ（破片）の軌道を計算しています。
    *   単なる二体問題だけでなく、**地球の扁平率による影響（J2摂動）**、**太陽や月による潮汐力（第三体摂動）**、**大気抗力（高度による密度変化と減速）**などをシミュレートしています。
*   **本格的な宇宙船の操作とナビゲーション**
    *   **RCS（姿勢制御システム）**による3軸姿勢制御（ピッチ・ヨー・ロール）および並進移動（前進・後退・左右・上下）。
    *   `[C]` キーによる**進行方向（Prograde）の自動維持（オートパイロット）**。
    *   画面下部中央に、宇宙船の姿勢を示す **ナビボール（Navball）** を描画。
*   **軌道計画マップモード (`[M]`キー)**
    *   地球を俯瞰するマップ画面に切り替えることができ、プレイヤーの軌道をクリックして**マニューバーノード（軌道変更計画）**を作成できます。
    *   WASDQEキーで必要速度変化（$\Delta v$）を調整し、軌道遷移の計画を立ててタイムワープで自動移動する機能があります。
*   **大気圏突入と熱シミュレーション**
    *   高度が下がると大気密度が上がって軌道が減衰します。
    *   大気摩擦によって機体が加熱され（サットン・グレイブス式）、放射冷却とのバランスで熱が計算されます。耐熱限界を超えると大気圏で燃え尽きてゲームオーバーになります。
*   **物理ベースのディテール**
    *   発砲すると薬莢が排出され、慣性で漂います。宇宙船が破壊されると破片が散らばり、非対称コマの特有の回転運動である **ジャニベコフ効果（テニスラケットの定理）** に基づいてスピンします。
    *   弾薬ベルトが宇宙船の挙動（推進加速度、回転時の遠心力・コリオリ力など）に応じてリアルに揺れる物理シミュレーション（Position-Based Dynamics）を搭載しています。
*   **WebAudioによるBGMと効果音**
    *   音声ファイルを読み込むのではなく、シンセサイザーの波形生成によってBGM（ミニマル・ミュージック風）やCIWSの射撃音・モーター音などをリアルタイム合成しています。

### 3. 特殊な描画アプローチ：「フローティング・オリジン (Floating Origin)」

広大な宇宙空間（地球低軌道）を float32（単精度浮動小数点数）で描画しようとすると、座標値が大きくなりすぎてジッター（グラフィックスのがたつき）が発生します。
このゲームでは、**プレイヤー宇宙船を常にワールド座標の原点 `(0, 0, 0)` に固定**し、地球や敵機などのすべてのオブジェクトをプレイヤーからの相対位置に毎フレーム配置し直す「フローティング・オリジン」アプローチを採用しています。これにより、カメラ付近の描画精度を極限まで保っています。

軌道計画モード(マップモード)の現状実装
概要
[M] キーでトグルする第2の操作モード。時間・物理シミュレーションは通常どおり進み続けたまま(simulate() は止まらない)、カメラと入力の解釈だけを「地球中心の俯瞰ビュー」に切り替え、将来のマニューバ(軌道変更)を視覚的に計画できるようにする機能です。実装は全部 game.ts 内に集約されています。

状態(フィールド)
mapMode: boolean — トグル状態。toggleMap()(case 'KeyM')で切替。
mapYaw/mapPitch/mapDist — 俯瞰カメラの球面座標(ドラッグ・ホイールで操作)。
mapCamera: THREE.PerspectiveCamera — 戦闘用カメラとは別インスタンス。near=1e4, far=6e8(モルニヤ級の高楕円軌道全体が収まる遠方)。
editNu: number | null — 編集中ノードの位置(自機軌道上の真近点角)。nullなら未配置。
editDv: Vec3 — 編集中のΔv(x=プログレード、y=ノーマル、z=ラジアルアウト成分)。
node: ManeuverNode | null — 確定済みの実行計画(nodeTime・rNode・vPlanned・targetEl・dvTotal)。
autoWarp: boolean — [N] によるノードへの自動タイムワープ中フラグ。
フロー
[M] でモード開始(toggleMap()) — mapMode=true、ヒントを表示。マップに入っても時間・物理は継続。
updateMapPlanning(dt)(mapMode中は毎フレーム呼ばれる)
現在軌道の要素を計算し、離心率0.98以上や非有限周期(双曲線軌道など)なら「計画できません」と表示して以降をスキップ。
クリックは pickOrbitNu() で処理: 自機軌道を離心近点角で512分割サンプルし、画面投影して最も近い点の真近点角を返す → editNu に設定。
editNu が設定済みなら、通常の推進キー(W/S/A/D/Q/E)をΔv調整に転用([V]微調整で感度をNODE_DV_RATE_FINEに切替)。
計画パネル(HUD左下)に Δv 内訳・到達までの時間・計画後の軌道要素(遠地点/近地点/傾斜角/周期)をリアルタイム表示(planHtml())。近地点が120km未満なら警告。
カメラ: mapYaw/mapPitch/mapDist をマウス/矢印キーで操作し、mapCamera を地球中心(フローティングオリジンなので -o)を注視する球面軌道カメラとして毎フレーム再配置。
再度 [M] で確定(finalizeNode()) — editNu・editDv から計画軌道要素を計算して this.node に凍結し、マップを抜ける。ΔvがNODE_MIN_DV未満なら何も確定しない。
[X] — マップ中なら編集中のノード(editNu/editDv)を破棄。戦闘中なら確定済みnodeを破棄。
[N] — nodeがあれば自動ワープをトグル。simulate()内で残り時間に応じて WARP_LEVELS から適切な段を自動選択し、AUTOWARP_STOP(20秒)前で解除。
戦闘ビューでの誘導(updateNodeGuide()) — ノード位置マーカー(残り時間カウントダウン付き)と、目標速度との差分方向を示す「BURN」マーカーを表示。
達成判定(orbitClose()) — 長半径の相対誤差・離心率差・軌道面角度差がすべて閾値内(NODE_TOL_SMA/ECC/PLANE_DEG)なら「マニューバ達成」としてノードを消去・ヒント表示。
表示面での特記事項
マップ中は方向系マーカー(プログレード/レトログレード/ボアサイト/リード)を隠し、自機位置マーカー(▷)のみ表示。
計画軌道線(白、plannedOrbitLine)は、マップ中はeditNuに基づくプレビュー、戦闘中は確定済みnode.targetElを表示。
マップ中のみ、生存中の全敵の軌道線(enemyOrbitLines)をターゲット以外も含めて表示し比較可能に。
既知の設計上の制約
双曲線軌道・離心率0.98以上・非有限周期の状態では計画不可(エラーメッセージのみ)。
ノード配置はスクリーン投影ベースのピクセル距離判定(NODE_PICK_PX)なので、軌道線が画面上で密集/重なる配置だと誤ってクリックしにくい。
finalizeNode() は現在の軌道要素を再計算するため、マップを開いてから閉じるまでの間に(自動ワープ等で)軌道が変化していても、確定時点の実軌道を基準に計画される(マップ表示中に見ていた軌道とズレる可能性はある設計)。

敵機のポリゴン削減 / LOD — 第零ステージで敵50機が5km圏内に密集する設計なので、遠距離の敵は低ポリ版に差し替える(または非表示化)のが最も効きそうです。ENEMY_SCALE=10の巨大メッシュが50体同時描画されるのは、モバイルGPUには重い可能性があります。
地球テクスチャの解像度を端末に応じて縮小 — 現状 2048×1024 の PNG を焼き出していますが、モバイル向けに1024×512版も書き出しておき、navigatorの画面サイズやメモリ推定(navigator.deviceMemory)で出し分ける、というのが手軽です。
パーティクル・デブリ・薬莢の上限を端末別に下げる — MAX_DEBRIS/MAX_CASINGS/MAX_BULLETS を定数でなく動的値にし、タッチデバイス検出時(TouchControls.isTouchDevice()が既にあります)は半分程度に絞る。
物理衝突判定のO(N²)の間引き — 現状は「N<200」前提のコメントがありますが、第零ステージ50機+ベルト12+薬莢260+デブリ160だと数百体規模になり得ます。距離の粗いバケット分割(グリッド)を入れるか、モバイルでは判定対象を間引く。
devicePixelRatioのキャップ — scene.ts で renderer.setPixelRatio(window.devicePixelRatio) をそのまま使っていますが、高密度スマホ(dpr 3〜4)はレンダリング解像度が跳ね上がるので Math.min(devicePixelRatio, 1.5〜2) 程度にキャップするのが、労力対効果が一番高いと思います。
まず着手するなら 5(dprキャップ) が数行の変更で最も費用対効果が高く、次に 1(敵のLOD/間引き) が第零ステージの体感負荷に直結すると思います。方向性が決まれば実装します。
